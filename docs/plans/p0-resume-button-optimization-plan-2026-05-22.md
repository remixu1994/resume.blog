# 简历页按钮设计与展示优化计划

- 计划日期：2026-05-22
- 页面范围：`/[locale]/resume`
- 负责人角色：产品/设计/前端（执行可由同一人兼任）
- 约束：本计划阶段不修改生产代码，仅用于规划与执行跟踪

## 1. 背景与目标

当前简历页首屏按钮存在可见的设计与展示问题，已影响页面专业感与可用性。目标是在不改变信息架构的前提下，完成按钮层面的可读性、可点击性、一致性与多端体验优化。

成功标准：
- 首屏按钮无空白/无文本状态
- 主次按钮层级清晰，视觉对比达标
- 桌面端与移动端均可稳定展示，交互状态完整
- 中英文文案长度变化下不破版

## 2. 已识别问题（基于截图 + 代码核对）

### P0（必须先修）

1. 次要按钮出现“空白小方块”
- 现象：主按钮右侧出现一个空白的次要按钮容器。
- 根因：联系按钮绑定字段 `contactEmail` 为空字符串（`src/legacy/content-data/src/index.ts` 中 `zh/en` 均为空）。
- 影响：用户无法识别按钮用途，且破坏首屏视觉完整性。

### P1（高优先）

2. 按钮语义与文案不一致
- 主按钮是“下载 PDF 简历”，次按钮实际是邮箱地址（技术字段）而非动作文案（如“联系我”）。
- 对非技术访客不友好，意图识别成本高。

3. 主次按钮样式层级弱
- 次按钮边框与背景对比偏弱，易被误判为占位元素。
- 两个按钮在视觉重量与间距上缺少统一规范（尺寸、最小宽度、文字对齐）。

### P2（中优先）

4. 状态设计不完整
- 需明确 `hover/focus/active/disabled` 状态，尤其是键盘焦点可见性。
- 缺少“数据缺失时”的降级方案（隐藏、替代文案或替代动作）。

5. 响应式与长文本风险
- 英文邮箱、长文案或移动端窄屏下可能挤压布局。
- 按钮最小触控面积与换行策略需明确。

## 3. 优化策略与决策

1. 信息策略
- 主按钮保持“下载简历”。
- 次按钮改为动作型文案（建议：“联系我”/“Email Me”），邮箱作为 `mailto` 目标而非按钮可见文本。

2. 数据策略
- 将 `contactEmail` 设为必填项（内容源层面）。
- 增加数据兜底：若邮箱缺失，次按钮不渲染或显示“暂不可用”且不可点击（二选一，建议不渲染）。

3. 视觉策略
- 建立按钮规范：高度、左右内边距、字重、圆角、边框与对比度阈值。
- 拉开主次层级：主按钮实色、次按钮浅底+清晰边框。
- 保证按钮组在桌面/移动端的对齐与间距一致。

4. 交互与可访问性策略
- 完整定义 `hover/focus-visible/active`。
- 确保键盘可访问、焦点清晰、可点击区域 >= 44px。

## 4. 执行优先级与排期（建议）

## 第 0 阶段（当天，P0）
- 修复内容源缺失：补齐 `zh/en contactEmail`。
- 决定缺失时兜底策略（推荐：不渲染次按钮）。
- 输出快速验收截图（桌面端）。

## 第 1 阶段（+1 天，P1）
- 统一按钮文案策略（动作文案替代原始邮箱字符串）。
- 制定按钮视觉规范（主次层级、尺寸、间距、对比度）。
- 补充状态样式定义。

## 第 2 阶段（+1~2 天，P2）
- 做响应式与多语言压测（中文/英文、长文案、窄屏）。
- 完成可访问性检查（键盘导航、焦点可见、屏幕阅读器标签）。

## 5. 验收清单（DoD）

- [ ] 主次按钮均有明确文本与动作，不出现空白容器
- [ ] `contactEmail` 在中英文内容源均非空
- [ ] 次按钮在邮箱缺失时按约定正确降级
- [ ] 桌面端（>=1024px）按钮组对齐正常
- [ ] 移动端（<=390px）按钮组不溢出、可点击
- [ ] `hover/focus-visible/active` 状态可见且一致
- [ ] Lighthouse / axe 基础可访问性无按钮相关阻塞项

## 6. 风险与依赖

- 依赖：内容维护流程是否允许“必填字段校验”。
- 风险：若继续将技术字段直接暴露为 UI 文案，后续多语言扩展会反复出现体验问题。
- 建议：把“内容完整性校验（如邮箱必填）”纳入发布前检查。
